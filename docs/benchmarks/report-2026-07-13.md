# Benchmark Report — 2026-07-13 (pre-optimization baseline)

**Machine:** MacBook Air, macOS 12, AppleClang 14, 4 cores  
**Build:** Release (`-O3` via CMake Release), C++20  
**Tool:** Google Benchmark v1.8.3 + custom percentile harness  
**Engine version:** v0.1.0 (post Phase 3, before any optimization)

## Throughput (mean time per operation)

|-----------|------|------------------|
| `BM_BestBid` | 2.6 ns | `map::begin()` — O(1) claim confirmed |
| `BM_AddOrder_ExistingLevel` | 360 ns | Add to a level that already exists (common case) |
| `BM_CancelOrder` | 621 ns | O(1) cancel via index (includes hash lookup + list erase) |
| `BM_Submit_RestNoMatch` | 651 ns | Full submit path, order rests (worst case: new level every time) |

## Latency percentiles (mixed workload, 100,000 ops, seed 42)

Workload: ~60% limit orders, ~20% market orders, ~20% cancels, prices in a
200-tick band (heavy matching pressure).

| Operation | n | p50 | p90 | p99 | p99.9 | max |
|-----------|---|-----|-----|-----|-------|-----|
| `submit_order` | 80,099 | 507 ns | 1,456 ns | 3,042 ns | 11,142 ns | 788,993 ns |
| `cancel_order` | 19,901 | 103 ns | 206 ns | 790 ns | 1,281 ns | 48,127 ns |

## Reading the numbers

- **p50 vs max for submit_order: 507ns vs 789μs — a 1,556x gap.** The
  median call is fast; the worst call is catastrophically slower. This is
  tail latency, and reducing it is the core discipline of low-latency work.
- Likely tail contributors (to be confirmed by profiling in Phase 4):
  - Heap allocations: every new `PriceLevel` (map node) and every
    `std::list` node is a `malloc` call; `std::vector<Trade>` growth
    reallocates
  - `std::list` cache misses: scattered nodes defeat the prefetcher
  - OS jitter: the process shares 4 cores with the rest of macOS; context
    switches land in the tail (unavoidable without core pinning)
- `BM_AddOrder_NewLevel` (804ns) vs `BM_AddOrder_ExistingLevel` (360ns):
  the ~450ns difference is roughly the cost of a red-black tree insert
  plus node allocation.

## Caveats

- Consumer laptop, no core isolation, background processes running —
  absolute numbers are indicative, not authoritative. Relative comparisons
  (before/after optimization) on the same machine are the meaningful signal.
- Numbers will regress if measured under thermal throttling; rerun on AC
  power with the machine idle.

## Next steps (Phase 4 optimization candidates)

1. Reserve `std::vector<Trade>` capacity in `submit_order`
2. Consider an object pool / arena for list nodes and levels
3. Re-measure and compare against this baseline

# Optimization 1: caller-provided Trade buffer (same day)

**Change:** added `submit_order(Order, std::vector<Trade>& out)` overload.
The caller owns and reuses the buffer; after warmup it never reallocates.
The convenience overload remains for cold paths (CLI, tests).

## Allocation counts (verified with mercury_alloc_count)

| Metric | Before | After |
|--------|--------|-------|
| Avg allocs per submit_order | 2.46 | **1.52** |
| Max allocs in one call | 7 | **4** |
| Zero-allocation calls | 0.04% | **46.8%** |

Remaining ~1.5 allocs per call: list node for resting orders, map node for
new price levels, hash map node for the order index. Candidate for a node
pool in a future pass.

## Measurement honesty note

Across three identical runs, max latency varied from 0.56 ms to 28.6 ms —
a 50x run-to-run swing caused by macOS scheduler preemption, not by engine
code. On shared consumer hardware, percentiles beyond p99 measure the
operating system, not the software under test. Extreme-tail claims would
require an isolated/pinned-core environment (or a Linux box with isolcpus).

**Conclusion:** the optimization reliably improved p50/p90/p99 and cut
allocations by ~40%, with 47% of submits now allocation-free. API safety
was preserved by making the caller own the buffer (no dangling-reference
hazard from returning an internal buffer).
