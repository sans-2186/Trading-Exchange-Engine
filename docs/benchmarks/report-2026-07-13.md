## Reading the numbers

- **p50 vs max for submit_order: 507ns vs 789ÎĽs â€” a 1,556x gap.** The
  median call is fast; the worst call is catastrophically slower. This is
  tail latency, and reducing it is the core discipline of low-latency work.
- Likely tail contributors (to be confirmed by profiling in Phase 4):
  - Heap allocations: every new `PriceLevel` (map node) and every
    `std::list` node is a `malloc` call; `std::vector<Trade>` growth
    reallocates
  - `std::list` cache misses: scattered nodes defeat the prefetcher
  - OS jitter: the process shares 4 cores with the rest of macOS; context
    switches land in the tail (unavoidable without core pinning)
- `BM_AddOrder_NewLevel` (804ns) vs `BM_AddOrder_ExistingLevel` (360ns):
  the ~450ns difference is roughly the cost of a red-black tree insert
  plus node allocation.

## Caveats

- Consumer laptop, no core isolation, background processes running â€”
  absolute numbers are indicative, not authoritative. Relative comparisons
  (before/after optimization) on the same machine are the meaningful signal.
- Numbers will regress if measured under thermal throttling; rerun on AC
  power with the machine idle.
